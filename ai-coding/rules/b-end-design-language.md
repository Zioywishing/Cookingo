# b-end-design-language.md

## 适用范围

当任务涉及 B 端后台页面、后台布局、后台组件库、后台表单、后台表格、后台弹窗、后台详情页、后台列表页时，必须优先遵守本规范。

本规范当前以 `docs/b-end/demo/demo.html` 作为唯一视觉基准，适用于 `app/layouts/admin.vue`、`app/pages/admin/**` 与 `app/components/admin/**`。

执行要求：

- 后台视觉语言统一以 `docs/b-end/demo/demo.html` 为准，不得在同一后台体系内混用现有深色玻璃风与新浅色风格。
- 后台组件新增或改造时，优先复用 `app/components/admin` 中的通用组件，不得在业务组件中重复散落按钮、卡片、表格、表单、弹窗样式。
- 页面级改造优先通过布局层、通用组件层与业务组合层收敛，不以页面内联样式堆叠为主要实现方式。
- 若后续设计语言与本规范不一致，必须先与人确认；确认后需要同步更新本规范。

## 1. 视觉基调

- 后台统一采用浅色画布、白色表面、轻边框、低阴影、粉紫渐变强调。
- 页面背景使用接近 `#F9FAFB` 的浅灰画布，主体卡片使用白色或接近白色表面。
- 边框使用低对比浅灰，避免深色描边或高饱和描边常驻。
- 品牌强调色仅用于主按钮、激活导航、品牌标识、关键强调信息，不大面积铺满内容背景。
- 阴影保持轻量，优先表达层级而不是炫光效果。

## 2. 字体系统

- 后台采用“衬线标题 + 无衬线正文”的双字体层级。
- 品牌名、页面主标题、关键统计数字优先使用带编辑感或出版感的衬线标题风格。
- 导航、表格、表单、按钮、辅助说明统一使用清晰的无衬线正文风格。
- 标题字号明显强于正文，但避免过度夸张；正文信息密度优先保证可读性与扫描效率。
- 小型标签、表头、分组标题允许使用全大写或增大字距来增强 B 端信息结构感。

## 3. 导航框架

- 后台统一采用“顶部横条 + 左侧导航 + 主内容区”的基础框架。
- 顶部横条负责品牌、当前用户信息与全局动作入口。
- 左侧导航负责模块分组、当前激活态与后台信息架构表达。
- 主内容区负责页面标题、描述、工具栏、卡片、表格与详情模块。
- 移动端保持相同设计语言，左侧导航收敛为抽屉，不另起一套移动端视觉系统。
- 导航激活态通过浅色高亮底、品牌色文字和左侧强调条表达，不使用厚重实色块。

## 4. 页面模板

- Dashboard 页面采用“标题区 + 统计概览 + 模块入口卡片”模板。
- List 页面采用“标题区 + 工具栏 + 表格卡片”模板。
- Detail 页面采用“标题区 + 多个信息卡片/配置卡片”模板。
- Auth 页面采用“聚焦单卡表单”模板，仍需与后台主站保持一致的颜色、字体、边框与按钮语言。
- Dialog 页面采用统一弹窗模板，保持一致的头部、内容区、底部动作区结构。
- 同类页面应复用同一模板，不为单页单独发明新的结构语言。

## 5. 组件状态规范

- 按钮必须具备 `default`、`hover`、`focus-visible`、`disabled`、`loading` 状态。
- 输入框与表单项必须具备 `default`、`focus`、`error`、`disabled` 状态。
- 表格必须具备 `loading`、`empty`、`data` 状态，并统一表头、行 hover、边框与滚动表现。
- 状态标签应统一映射 `active`、`pending`、`disabled`、`danger` 等业务状态，不允许每个业务组件自定义一套颜色表达。
- 卡片、导航项、弹窗、可点击模块卡片都必须提供清晰但克制的 hover 与 focus-visible 反馈。
- 所有交互组件都必须保证键盘可访问性与可见焦点。

## 组件实现约束

- 后台通用组件统一维护在 `app/components/admin` 下，不新增脱离 Nuxt 约定的新前端目录。
- 推荐将后台组件拆分为框架层、通用基础层、业务组合层。
- 通用基础层优先承接卡片、按钮、输入框、表格、弹窗、状态标签、空状态、工具栏等通用能力。
- 业务层组件只负责业务字段映射与局部交互，不重复维护通用样式规范。

## 禁止事项

- 禁止在后台页面中继续引入深色玻璃拟态背景、重阴影、蓝青色霓虹高光等与本规范冲突的风格。
- 禁止在不同后台页面中混用两套按钮、表格、输入框或弹窗样式。
- 禁止将通用 B 端设计 token 与样式散落在多个页面文件中长期维护。
- 禁止为了局部页面临时新增语义不清晰的后台组件目录。

## 例外处理

- 若某个后台页面因安全告警、危险操作需要更强提示，可以在本规范基础上局部增强危险状态，但不得脱离整体视觉语言。
- 若用户明确指定某个后台页面需偏离本规范，以用户要求为准，同时应评估是否更新本规范。
