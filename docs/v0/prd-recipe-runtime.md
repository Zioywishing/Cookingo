# PRD: `IRecipeProcessStack` Runtime

## 1. 背景

`IRecipeProcessStack` 定义如下：

```ts
export type IRecipeProcessStack = IRecipeProcessNode[];
```

在 runtime 语义下，它应被视为一个会被持续修改的工作栈：

- 栈顶节点，代表当前唯一展示给用户的步骤。
- 用户完成的，总是当前栈顶节点。
- runtime 通过 `pop`、`push`、挂起、恢复，持续重排这个栈。

这里的核心是：始终为用户提供一个明确的当前步骤，同时保留多链流程、依赖汇合和异步等待能力。

## 2. 目标

本 runtime 文档只定义单个 `IRecipeProcessStack` 的产品行为模型，目标是：

- 明确 `IRecipeProcessStack` 在运行时的真实含义。
- 明确“栈顶即当前步骤”的用户体验约束。
- 明确 `next`、`prev`、`async` 在栈模型中的调度语义。
- 明确节点完成、等待结束、依赖满足时，stack 应如何变化。
- 为后续实现提供统一的调度原则，但不绑定具体 API。

## 3. 核心结论

### 3.1 `IRecipeProcessStack` 的 runtime 含义

`IRecipeProcessStack` 在 runtime 中表示“当前待执行工作栈”。

- 数组顶部元素，即当前步骤。
- 数组中更靠下的元素，表示后续候选步骤或暂时排队中的流程链。
- runtime 会在执行过程中动态修改这个数组内容与顺序。

因此，它是调度器本身正在维护的运行中栈状态。

### 3.2 用户体验原则

产品层必须满足一个强约束：

- 任意时刻，只向用户展示栈顶节点。

这意味着 runtime 需要持续算出“现在栈顶应该是谁”，并把它作为当前步骤展示给用户。

## 4. 基础调度模型

### 4.1 栈顶规则

设当前 runtime stack 为：

```ts
[A, B, C]
```

则：

- `A` 为栈顶。
- 当前用户只看到 `A`。
- `B`、`C` 暂不展示，但仍属于运行中的待执行内容。

### 4.2 节点完成规则

当用户完成当前栈顶节点时，runtime 必须先将该节点出栈。

即：

- 当前栈顶完成，先 `pop`。
- 然后再根据该节点的 `next`、`async`、以及其他节点的 `prev` 满足情况，决定是否有新节点需要 `push` 回栈顶。

这条规则非常关键，顺序如下：

1. 先移除当前节点。
2. 再推进后续节点。

## 5. `next` 的语义

### 5.1 普通顺序推进

若当前栈顶节点 `A` 没有 `async`，且存在 `next = B`，则当 `A` 完成后：

1. `A` 出栈。
2. 检查 `B` 是否允许进入栈。
3. 若允许，则将 `B` 压入栈顶。

结果是用户自然从 `A` 进入 `B`。

### 5.2 `next` 不等于无条件压栈

`next` 表示“默认后继方向”。

如果 `B` 还带有 `prev`，那么 runtime 必须先检查：

- `B.prev` 中的所有前置节点是否已经完成。

只有全部完成，`B` 才进入栈顶。

否则：

- `A` 仍然正常出栈。
- `B` 保持未入栈状态。
- stack 继续显示当前剩余栈顶，直到未来某个时刻 `B` 被解锁。

## 6. `async` 的语义

### 6.1 `async` 的本质

`async.waitMinutes` 表示：

- 当前节点的“动手动作”完成后，当前节点先从工作栈中移出。
- 等待结束后，再把它的 `next` 重新压回栈顶。

这是 `recipe.ts` 注释中的核心约束。

### 6.2 `async` 节点完成时的行为

若当前栈顶节点 `A` 带有：

```ts
async: { waitMinutes: 20 }
```

且 `A.next = B`，则当用户完成 `A` 的手头动作时：

1. `A` 立即从栈顶出栈。
2. runtime 记录一个等待中的挂起任务。
3. `B` 保持未入栈状态。
4. 用户界面立即切换到新的栈顶节点。

也就是说，`async` 的效果是“当前链临时让出栈顶，让用户去做别的事”。

### 6.3 `async` 等待结束时的行为

当等待时间结束后：

1. runtime 将该异步节点视为正式完成。
2. 检查它的 `next` 是否满足执行条件。
3. 若满足，则将 `next` 压回栈顶。

这里采用“压回栈顶”的方式，使被挂起后恢复的流程线在等待结束后立即重新成为用户当前步骤。

### 6.4 `async` 无 `next`

若异步节点没有 `next`，则含义为：

- 它在等待结束后，这条链彻底完成。
- runtime 结束挂起记录。

## 7. `prev` 的语义

### 7.1 `prev` 是执行前置条件

`prev` 表达的是：

- 这个节点位于栈顶时，是否允许用户执行。

例如：

```ts
D.prev = [A, B, C]
```

其含义是：

- 只有当 `A`、`B`、`C` 全部完成，`D` 到达栈顶后才允许执行。

### 7.2 `prev` 是 AND 依赖

`prev` 按“全部满足”理解。

只要 `prev` 中还有任一节点未完成，当前栈顶节点就处于阻塞状态。

### 7.3 `prev` 的阻塞行为

当一个带 `prev` 的节点来到栈顶时，runtime 立即检查它的前置条件。

该文档约定如下：

- 若 `prev` 已全部满足，则该节点可正常执行。
- 若 `prev` 未全部满足，则当前流程直接阻塞在这个栈顶节点。

原因是：

- stack 本身已经表达了展示顺序。
- `prev` 负责校验当前步骤能否被执行。
- 当栈顶步骤的前置尚未完成时，用户当前就不能继续推进。

## 8. 初始化规则

### 8.1 初始 stack 的来源

recipe schema 中写下来的 `IRecipeProcessStack` 是初始工作栈。

其数组顺序本身具有产品含义：

- 越靠前的节点，越优先展示给用户。
- 若多个链头在开始时都可执行，则第一个元素作为初始栈顶。

因此，编排者在写 stack 时，不只是声明“有哪些头节点”，也同时声明“它们的初始优先级”。

### 8.2 启动时的展示方式

runtime 启动时：

- 直接以当前数组第一个元素作为栈顶展示。
- 其他链头保留在 stack 中，作为后续待执行内容存在。

## 9. 最小调度规则

### 9.1 当前栈顶为普通节点时

当栈顶节点完成：

1. 当前节点出栈。
2. 若存在满足条件的 `next`，压入栈顶。
3. 用户看到新的栈顶。

### 9.2 当前栈顶带有 `prev` 时

当栈顶节点声明了 `prev`：

1. runtime 检查 `prev` 中的节点是否已全部完成。
2. 若已全部完成，则该节点可执行。
3. 若仍有未完成前置，则当前流程阻塞在这个栈顶节点。

### 9.3 当前栈顶为异步节点时

当栈顶异步节点完成手头动作：

1. 当前节点出栈。
2. 记录等待任务。
3. 不立即推进它的 `next`。
4. 继续显示新的栈顶。
5. 等等待结束后，再把它的 `next` 压回栈顶。

### 9.4 等待结束时

当某个挂起任务到时：

1. 将其对应链视为恢复。
2. 若其 `next` 满足条件，则将 `next` 压回栈顶。
3. 当前用户界面立即切换到新的栈顶。

## 10. 典型示例

给定以下流程：

- 切番茄
- 打蛋液
- 切葱花
- 炒鸡蛋，`prev: [切番茄, 打蛋液, 切葱花]`
- 炒番茄
- 合炒出锅

其中初始 stack 为：

```ts
[切番茄, 打蛋液, 切葱花, 炒鸡蛋]
```

执行过程如下：

1. 用户先看到栈顶 `切番茄`。
2. `切番茄` 完成后出栈，栈顶变为 `打蛋液`。
3. `打蛋液` 完成后出栈，栈顶变为 `切葱花`。
4. `切葱花` 完成后出栈，栈顶变为 `炒鸡蛋`。
5. `炒鸡蛋` 来到栈顶时，runtime 检查它的 `prev`。
6. 由于 `切番茄`、`打蛋液`、`切葱花` 都已完成，`炒鸡蛋` 可执行。
7. `炒鸡蛋` 完成后出栈，并把 `炒番茄` 压回栈顶。
8. `炒番茄` 完成后，把 `合炒出锅` 压回栈顶。
9. `合炒出锅` 完成后，整个 stack 完成。

这个示例体现了 4 个关键事实：

- 用户始终只看一个当前步骤。
- 初始数组顺序决定开始时的展示顺序。
- `prev` 在节点到达栈顶时被校验。
- `prev` 未满足时，流程会阻塞在当前栈顶。

## 11. 数据约束

为了让该 runtime 可实现，`IRecipeProcessStack` 应满足以下约束：

- stack 中元素是各条链当前的合法起点。
- `next` 形成无环推进关系。
- `prev` 形成可校验的前置关系。
- 一个节点若声明了 `prev`，这些前置节点必须在同一运行语义范围内可被追踪完成。
- `async.waitMinutes` 必须为正数。

## 12. 结论

`IRecipeProcessStack` 的 runtime 本质上是一个“单可见栈顶”的调度器。

它的核心原则只有五条：

- stack 是运行中的工作栈。
- 用户始终只看栈顶节点。
- 当前节点完成后先出栈，再决定后续压栈。
- `async` 的语义是“先出栈，等待后恢复下一步到栈顶”。
- `prev` 的语义是“节点位于栈顶时的执行前置条件”。

任何具体实现，只要满足这五条，就可以认为兼容 `IRecipeProcessStack` 的 runtime 产品模型。
