# Cookingo Default Layout 主题壳层改造方案

## 1. 文档目标

本文档用于定义 Cookingo 前台默认布局的主题壳层改造方案。

本次方案聚焦 3 个目标：

- 将首页当前的背景壳层能力上移到 `default layout`
- 将主题色实现改为可复用的全局语义 CSS 变量注入
- 为后续默认布局页面复用同一背景和页面切换动画预留结构

本次方案明确要求：

- 首页当前视觉效果不发生变化
- 主题色仍按固定 `UTC+8` 时间段切换
- `admin` 路由不得注入这套前台主题

## 2. 当前现状

### 2.1 首页当前实现

当前首页位于 `app/pages/index.vue`，同时承担了以下职责：

- 调用 `useHomepageTheme()` 获取时间段主题状态
- 通过内联 `style` 注入一组 `--homepage-*` CSS 变量
- 自身承载外层背景、内层 frame、ambient 光晕层
- 渲染首页内容区域（状态 pill、logo、文案、CTA）

结果是：主题系统、背景系统、页面内容系统都耦合在首页页面文件中。

### 2.2 默认布局当前实现

当前 `app/layouts/default.vue` 只是一个通用滚动容器：

- 没有主题能力
- 没有视觉壳层
- 没有页面内容层的独立语义容器

这意味着未来其他前台页面如果要复用首页背景，只能复制首页壳层或抽一个页面级背景组件，无法通过布局统一承接。

### 2.3 当前方案的主要问题

- 背景壳层无法被默认布局下的其他页面自然复用
- 主题变量命名带有明显首页语义，不适合作为前台通用主题系统
- 页面通过内联样式注入变量，后续扩展到多页面时归属不清晰
- 若后续要做内容视图切换动画，当前背景和内容没有被清晰拆层

## 3. 已确认约束

本次方案基于以下已确认结论：

- 首页视觉效果必须保持不变
- 默认布局下的后续页面需要复用同一套背景能力
- 主题配色继续按 `UTC+8` 当前时间段切换
- 主题实现改为 CSS 变量注入，页面和组件只负责消费变量
- 主题颜色切换需要有平滑过渡
- 本次不实现页面进入退出动画，只预留结构
- `admin` 路由不应注入这套前台主题

## 4. 设计决策

### 4.1 将背景壳层提升到 `default layout`

`app/layouts/default.vue` 升级为前台默认视觉壳层，负责：

- 承接主题状态与主题变量注入
- 渲染外层背景 shell
- 渲染内层 frame
- 渲染 ambient 光晕层
- 提供独立的 `view` 内容容器包裹 `<slot />`

这样后续所有使用默认布局的前台页面，都可以天然复用同一套背景与主题。

### 4.2 主题变量注入到 `body`，而不是停留在页面或 layout 根节点

本次主题变量不再由首页页面通过内联样式挂在自身根节点，而是由默认布局通过 `useHead()` 注入到 `body`。

选择 `body` 注入而不是 layout 根节点，原因如下：

- 主题变量可作为前台全局语义变量使用，而不是“某个页面容器里的局部变量”
- 默认布局树内所有页面与组件都可以直接消费，不需要依赖中间包裹层
- SSR 首屏可以直接输出正确变量，减少首屏颜色抖动风险
- 后续如需为页面切换动画引入额外的内容包裹层，不会影响主题变量作用域

### 4.3 `admin layout` 显式隔离前台主题

虽然主题变量注入目标是 `body`，但本次不会把它做成“全站统一总是存在”的样式。

边界定义如下：

- `default layout` 负责设置前台主题 `body` 标记并注入前台主题变量
- `admin layout` 负责显式设置自己的 `body` 标记，不注入前台主题变量
- layout 切换时，前台主题变量不得在 `body` 上残留到 `admin` 场景

这意味着：

- 默认布局下的页面获得前台主题
- `/admin/**` 页面不会意外消费到前台主题
- 前后台切路由时，`body` 上的主题语义是显式切换，而不是依赖“残留状态刚好被覆盖”

### 4.4 主题变量命名改为全局前台语义

当前 `--homepage-*` 命名不再适合作为目标形态。

本次改造后，变量命名应按“前台默认主题”语义设计，而不是首页专用语义。建议统一使用 `--app-theme-*` 前缀，例如：

- `--app-theme-shell-background`
- `--app-theme-surface-base`
- `--app-theme-surface-elevated`
- `--app-theme-text-primary`
- `--app-theme-text-secondary`
- `--app-theme-accent`
- `--app-theme-accent-hover`
- `--app-theme-glow`
- `--app-theme-action-primary`
- `--app-theme-action-primary-text`

命名规则：

- 面向消费语义，而不是来源页面
- 尽量表达用途，而不是具体色值位置
- 保持后续其他页面、组件可直接复用

### 4.5 主题颜色过渡在变量层完成

本次不要求每个消费组件单独添加一套 `color/background/border-color` 过渡规则来响应时间段切换。

推荐方案是：

- 为颜色类主题变量注册可动画的 CSS 自定义属性
- 在主题变量发生变化时，由变量层统一完成颜色过渡

这样可以保证所有消费这些变量的前台元素自动获得平滑过渡，包括：

- 文字颜色
- 背景颜色
- 描边与边框颜色
- 阴影中的主题色
- 光晕和渐变中的主题色

同时，本次只处理“主题变色过渡”，不把页面内容自己的进入动画绑进同一套机制。

### 4.6 首页页面只保留内容视图职责

`app/pages/index.vue` 在改造后只保留首页内容层：

- 时间状态展示
- logo
- 标题与文案
- CTA
- 当前内容区块的入场动画

首页不再承接：

- 外层主题变量注入
- 背景 shell
- frame
- ambient 光晕层

这样首页成为纯粹的 view 页面，布局负责壳层，职责边界更清晰。

## 5. 主题系统设计

### 5.1 保留时间段与文案算法

当前 `UTC+8` 时间段划分逻辑和文案策略继续保留，不改变以下行为：

- 时间段划分区间
- 首屏固定时间基准
- 当前时间文本显示
- 文案轮换逻辑

原因是本次目标是抽离主题承载方式，而不是重新定义主题算法。

### 5.2 状态层与变量层分离

当前 `useHomepageTheme()` 中“主题状态”和“页面样式变量”混在一起。

改造后的目标是拆成两个清晰出口：

- 主题状态：供页面文案和时间信息展示使用
- 主题变量：供布局注入到 `body`，再由 CSS 消费

状态层应继续包含：

- `segment`
- `label`
- `timeRange`
- `headline`
- `currentTimeLabel`

变量层应只负责返回全局语义变量值映射，而不再以 `heroStyle` 这类首页视图概念命名。

### 5.3 SSR 约束

为避免服务端与客户端首屏主题不一致，本次继续保留当前“首屏固定时间戳 + 客户端按分钟更新”的思路。

预期行为：

- SSR 输出时，默认布局使用同一基准时间计算主题变量
- hydration 后不应立即产生无意义的颜色闪动
- 客户端进入新的分钟或新的时间段后，再更新主题状态和变量

### 5.4 Body 注入结构

默认布局应通过 `useHead()` 输出以下内容：

- `bodyAttrs`
- 主题标记
- 主题变量

建议 body 标记语义：

- 默认布局：`data-app-theme="default"`
- admin 布局：`data-app-theme="admin"`

变量注入方式应保持实现可维护，不建议把大量字符串拼接散落在页面中。

同时需要保证：

- 默认布局卸载或切换到 `admin layout` 时，`body` 上旧的 `--app-theme-*` 变量会被正确移除或替换
- `body` 级主题注入不会污染 `admin layout` 已有的 `--admin-*` 变量系统

## 6. 布局与页面结构设计

### 6.1 `default layout`

默认布局目标结构：

```vue
<div class="default-layout">
  <div class="default-layout__frame">
    <div class="default-layout__ambient" aria-hidden="true">
      ...
    </div>

    <div class="default-layout__view">
      <slot />
    </div>
  </div>
</div>
```

职责说明：

- `default-layout`：承载全屏背景、safe-area、滚动与根级语义
- `default-layout__frame`：承接当前首页卡片式容器的尺寸、底色、圆角和阴影
- `default-layout__ambient`：承接两个动态光晕层
- `default-layout__view`：承接页面内容，后续这里接路由进入退出动画

### 6.2 `index.vue`

首页页面在改造后只保留内容结构，不再拥有背景壳层节点。

页面内仍可展示时间段标签、当前时间、文案等，但这些数据来源于共享主题状态，而不是页面自己重新维护一份颜色系统。

### 6.3 后续页面复用方式

后续任何使用 `default layout` 的前台页面都默认拥有：

- 同一套时间主题变量
- 同一套背景 shell
- 同一层 frame 容器
- 同一层 ambient 动态背景

页面自身只需要在内容样式中消费全局变量即可。

## 7. 样式迁移边界

### 7.1 迁移到 `default layout` 的内容

应迁移的内容包括：

- 当前首页外层全屏背景与 safe-area padding
- frame 的尺寸、圆角、阴影、底色
- ambient 光晕层的定位、模糊与动画
- 默认布局下的主内容容器层

### 7.2 保留在 `index.vue` 的内容

应保留在首页页面内的内容包括：

- 状态 pill
- logo
- eyebrow / brand / welcome
- CTA
- 首页内容本身的入场动画

### 7.3 视觉不变原则

本次改造必须遵守“视觉不变优先于结构好看”的原则。

只允许改变：

- DOM 承载位置
- 主题变量命名
- 主题注入位置

不允许主动改变：

- 首页 frame 的尺寸与比例
- 光晕层的位置和动态节奏
- 首页文案布局节奏
- 当前首页按钮、文本、logo 的视觉关系

## 8. 路由动画预留设计

本次不实现路由进入退出动画，但需要保证结构已满足后续接入条件。

约束如下：

- `default-layout__view` 必须是一个独立内容层
- 背景层与内容层必须清晰分离
- 后续如果在 `view` 层外包 `Transition` 或 `TransitionGroup`，不应影响背景层稳定存在

这意味着未来页面切换时，只切内容视图，不重新创建背景壳层。

## 9. 文件改动范围

### 9.1 布局层

- Modify: `app/layouts/default.vue`
- Modify: `app/layouts/admin.vue`

### 9.2 主题逻辑层

- Modify: `app/composables/useHomepageTheme.ts`

说明：

- 文件名是否继续保留 `useHomepageTheme.ts`，取决于实现阶段是否顺手完成语义升级
- 如果仅修改内部职责、不影响复用判断，本次可先保留原文件名做最小改动
- 如果实现时发现名称已经明显误导职责，则应在实现计划阶段明确是否重命名为更通用的前台主题 composable

### 9.3 页面层

- Modify: `app/pages/index.vue`

### 9.4 样式载体

- Modify: 现有承载默认布局样式的文件
- 如实现阶段需要为 `@property` 注册创建明确的全局样式承载位置，可在不破坏现有目录语义的前提下做最小新增

## 10. 错误处理与兼容性

### 10.1 主题计算异常

若主题状态计算出现异常，最低要求是前台页面仍能渲染，而不是因为主题系统失败导致默认布局不可用。

因此实现阶段需要确保：

- 变量映射始终有默认值
- `body` 注入内容不会因单个字段为空而中断

### 10.2 浏览器兼容性

可动画 CSS 变量依赖 `@property` 支持。

本次兼容性策略应明确为：

- 支持 `@property` 的环境获得平滑主题过渡
- 不支持的环境至少仍能正确切换主题色，不允许功能失效

也就是说，过渡能力是增强项，主题正确性是基础项。

## 11. 测试与验证

### 11.1 视觉验证

必须验证以下结果：

- 首页首屏视觉与改造前保持一致
- 不同时段主题切换后，背景、文本、CTA、光晕颜色仍正确
- 默认布局的 frame 与安全区表现不偏移

### 11.2 结构验证

必须验证以下结果：

- 默认布局已形成“背景层 + view 层”的清晰分层
- 首页不再承接背景壳层职责
- 默认布局下其他页面具备直接复用主题与背景的条件

### 11.3 路由隔离验证

必须验证以下结果：

- `default layout` 页面能在 `body` 上看到前台主题标记与变量
- `admin layout` 页面不会注入这套前台主题
- 从前台切到 admin、再切回前台时，不出现主题标记或 `--app-theme-*` 变量残留导致的样式串扰

### 11.4 质量门槛

实现阶段完成后，必须执行：

```bash
bun quality-check
```

并且不允许出现任何报错。

## 12. 非目标

本次方案不包括：

- 新增或重构页面进入退出动画实现
- 调整首页文案策略
- 调整时间段划分规则
- 为 admin 主题引入同一套时间色系统
- 对与本次主题壳层改造无关的前台页面做无关重构
