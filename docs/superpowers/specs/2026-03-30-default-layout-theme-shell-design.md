# Cookingo Default Layout 主题壳层改造方案

## 1. 文档目标

本文档用于定义 Cookingo 前台默认布局的主题壳层改造方案。

本次方案聚焦以下目标：

- 将首页当前的背景壳层能力上移到 `default layout`
- 将主题色系统统一收口到 `useGlobalTheme`
- 让前台主题在 SSR 输出的页面中通过前置脚本完成注入，并基于用户本地时间生效
- 保持首页当前视觉效果不变
- 为后续默认布局页面复用同一背景与主题系统预留结构

本次方案明确要求：

- `default layout` 只负责消费 CSS 变量，不负责主题计算与注入
- `admin` 路由不能受到这套前台主题注入的任何影响
- 主题只提供颜色，不负责动画与过渡
- 颜色过渡由消费模板的 class 样式自行承担

## 2. 当前现状

### 2.1 首页当前实现

当前首页位于 `app/pages/index.vue`，同时承担了以下职责：

- 调用 `useHomepageTheme()` 获取时间段主题状态
- 通过内联 `style` 注入一组 `--homepage-*` CSS 变量
- 自身承载外层背景、内层 frame、ambient 光晕层
- 渲染首页内容区域（状态 pill、logo、文案、CTA）

结果是：主题系统、背景系统、页面内容系统都耦合在首页页面文件中。

### 2.2 默认布局当前实现

当前 `app/layouts/default.vue` 只是一个通用滚动容器：

- 没有背景壳层
- 没有统一主题消费结构
- 没有与背景层分离的独立内容层语义

这意味着未来其他前台页面如果要复用首页背景，只能复制首页壳层或自行包裹背景组件，无法通过默认布局天然获得。

### 2.3 当前方案的主要问题

- 主题色变量命名带有明显首页语义，不适合作为前台全局主题系统
- 页面内联注入变量的方式不利于抽离成独立主题入口
- 背景与内容没有清晰拆层，不利于后续做内容视图切换动画
- 现有主题计算把“文案状态”和“颜色主题”混在一起，不利于职责分离

## 3. 已确认约束

本次方案基于以下已确认结论：

- 首页视觉效果必须保持不变
- 默认布局下的后续页面需要复用同一套背景能力
- 主题配色应按用户本地时间切换，不再固定 `UTC+8`
- `useGlobalTheme` 是唯一主题入口
- SSR 阶段不直接计算主题颜色，而是输出前置脚本，让浏览器在 hydration 前按本地时间完成主题注入
- `default layout` 只负责消费变量，不负责主题逻辑
- `admin` 路由不输出这套前台主题注入脚本
- 主题层只提供颜色，颜色过渡由模板样式层承担
- 本次不实现页面进入退出动画，只预留结构

## 4. 设计决策

### 4.1 将背景壳层提升到 `default layout`

`app/layouts/default.vue` 升级为前台默认视觉壳层，负责：

- 渲染外层背景 shell
- 渲染内层 frame
- 渲染 ambient 光晕层
- 提供独立的 `view` 内容容器包裹 `<slot />`

默认布局不再承担主题计算、时间判断、脚本注入等职责，只消费已经注入完成的 `--app-theme-*` 变量。

### 4.2 `useGlobalTheme` 统一收口主题系统

本次改造后，前台主题系统统一由 `useGlobalTheme` 承接。

`useGlobalTheme` 的职责应严格限定为：

- 定义时间段到颜色主题的映射关系
- 生成主题变量映射
- 生成 SSR 输出用的前置注入脚本
- 提供运行时需要的最小主题读取能力

`useGlobalTheme` 不负责：

- 首页文案
- 颜色过渡动画
- 布局渲染
- 页面进入退出动画

### 4.3 主题注入承载点放在非视觉入口层，而不是 layout

由于 `default layout` 只负责消费变量，因此本次主题注入不应写在 layout 中。

推荐承载点为 `app/app.vue` 这一非视觉入口层：

- 根据当前路由或 layout 语义判断当前页面是否属于前台默认布局
- 仅在前台默认布局页面输出 `useGlobalTheme` 生成的前置脚本
- `admin` 页面不输出这段脚本

这样可以同时满足：

- 主题注入与视觉布局解耦
- 默认布局页面天然拥有主题注入能力
- `admin` 路由完全隔离，不走这套主题链路

### 4.4 主题变量命名改为全局前台语义

当前 `--homepage-*` 命名不再适合作为目标形态。

本次改造后，变量命名应按“前台全局主题”语义设计，而不是首页专用语义。建议统一使用 `--app-theme-*` 前缀，例如：

- `--app-theme-shell-background`
- `--app-theme-surface-base`
- `--app-theme-surface-elevated`
- `--app-theme-text-primary`
- `--app-theme-text-secondary`
- `--app-theme-accent`
- `--app-theme-accent-hover`
- `--app-theme-glow`
- `--app-theme-action-primary`
- `--app-theme-action-primary-text`

命名规则：

- 面向消费语义，而不是来源页面
- 面向颜色用途，而不是视觉区块历史名字
- 保持后续前台其他页面、组件可直接复用

### 4.5 主题时间基于用户本地时间

本次主题不再依赖固定 `UTC+8`。

主题时间来源改为浏览器本地时间，原因如下：

- 本次主题颜色选择在浏览器端前置脚本中完成
- 不再依赖服务端推断用户时区
- 可以避免“服务端时间语义”和“用户真实本地时间语义”不一致

这意味着：

- SSR 阶段只负责输出脚本，不负责决定最终主题颜色
- 主题颜色在浏览器端基于 `new Date()` 的本地时间生成

### 4.6 主题过渡放回消费模板层

本次不在 theme 层做颜色变量动画。

主题层只提供静态颜色值映射，实际的颜色过渡由消费样式自行声明，例如：

- `color`
- `background-color`
- `border-color`
- `box-shadow`
- `stroke`
- `fill`

这样做的原因是：

- `useGlobalTheme` 保持纯粹，只负责“给什么颜色”
- 视觉元素可以按自身需要决定是否需要过渡、过渡哪些属性、过渡时长和节奏
- 避免把动画策略固化进主题系统

### 4.7 首页页面只保留内容视图职责

`app/pages/index.vue` 在改造后只保留首页内容层：

- 时间状态展示
- logo
- 标题与文案
- CTA
- 内容区块的入场动画

首页不再承接：

- 全局主题变量注入
- 背景 shell
- frame
- ambient 光晕层

## 5. 主题系统设计

### 5.1 `useGlobalTheme` 的输出形态

`useGlobalTheme` 应至少提供两类能力：

- 主题变量映射能力
- SSR 前置脚本生成能力

主题变量映射能力用于：

- 根据时间段计算出应使用的 `--app-theme-*` 变量值

SSR 前置脚本生成能力用于：

- 在浏览器 hydration 前运行
- 读取浏览器本地时间
- 计算当前时间段对应的主题色
- 将 `--app-theme-*` 写入页面

### 5.2 SSR 前置脚本策略

本次 SSR 相关逻辑的重点不是“服务端直接输出颜色”，而是“服务端输出一段足够早执行的脚本”。

预期流程如下：

1. 服务端渲染默认布局页面
2. 非视觉入口层输出 `useGlobalTheme` 生成的前置脚本
3. 浏览器在 hydration 前执行脚本
4. 脚本基于本地时间计算主题
5. 脚本将 `--app-theme-*` 注入页面
6. `default layout` 与页面内容直接消费这些变量

这个流程的意义是：

- 不依赖服务端知道用户时区
- 不要求 mounted 后再切主题
- 首屏主题由浏览器本地时间决定

### 5.3 注入目标

主题变量需要注入到一个足以被默认布局及其子页面共同消费、但又不应由 layout 自己负责维护的宿主上。

设计要求如下：

- 宿主必须能被默认布局和页面内容直接访问到
- 宿主不应要求每个页面单独重复注入
- 宿主选择应与 `app/app.vue` 这类非视觉入口层协调

具体注入到 `html`、`body` 或其他顶层宿主的实现细节，可在实现计划阶段确定，但必须满足以下结果：

- 默认布局中的背景和内容都能直接消费变量
- `admin` 页面不会输出这套前台主题注入
- 主题注入逻辑不依附 `default layout`

### 5.4 主题只提供颜色，不提供文案

现有 `useHomepageTheme()` 同时提供：

- 调色板
- 时间标签
- 时间段文案
- 当前时间文本

改造后需要拆责：

- `useGlobalTheme` 只负责颜色主题
- 首页文案与时间展示逻辑应保留为首页专属能力，不继续耦合在全局主题入口中

实现阶段可以选择：

- 保留现有首页专属 composable，但移除其颜色职责
- 或拆成更清晰的首页内容 composable

无论采用哪种方式，都必须保证 `useGlobalTheme` 不继续承载首页专属文案逻辑。

## 6. 布局与页面结构设计

### 6.1 `default layout`

默认布局目标结构：

```vue
<div class="default-layout">
  <div class="default-layout__frame">
    <div class="default-layout__ambient" aria-hidden="true">
      ...
    </div>

    <div class="default-layout__view">
      <slot />
    </div>
  </div>
</div>
```

职责说明：

- `default-layout`：承载全屏背景、safe-area、滚动与根级布局语义
- `default-layout__frame`：承接当前首页卡片式容器的尺寸、底色、圆角和阴影
- `default-layout__ambient`：承接两个动态光晕层
- `default-layout__view`：承接页面内容，后续这里接路由进入退出动画

布局中所有颜色相关样式都应改为消费 `var(--app-theme-*)`。

### 6.2 `index.vue`

首页页面在改造后只保留内容结构，不再拥有背景壳层节点。

页面仍可展示时间状态、标题、文案和 CTA，但这些内容不应再承担全局主题注入职责。

### 6.3 后续页面复用方式

后续任何使用默认布局的前台页面，都应天然获得：

- 同一套背景 shell
- 同一套 `--app-theme-*` 变量消费能力
- 同一层 frame 容器
- 同一层 ambient 动态背景

页面自身只负责在内容样式中消费变量。

## 7. 样式迁移边界

### 7.1 迁移到 `default layout` 的内容

应迁移的内容包括：

- 当前首页外层全屏背景与 safe-area padding
- frame 的尺寸、圆角、阴影、底色
- ambient 光晕层的定位、模糊与动画
- 默认布局下的主内容容器层

### 7.2 保留在 `index.vue` 的内容

应保留在首页页面内的内容包括：

- 状态 pill
- logo
- eyebrow / brand / welcome
- CTA
- 首页内容本身的入场动画

### 7.3 颜色过渡放在消费样式层

本次改造后，凡是直接消费主题变量且需要平滑切色的元素，都应在自身 class 样式中显式声明过渡。

重点覆盖对象包括：

- 默认布局根层与 frame
- 光晕相关样式
- 首页文案文字
- 首页按钮
- 图标描边与强调色元素

这里的目标是：

- 主题变化时视觉平滑
- 过渡策略归属于消费者，而不是主题入口

## 8. 路由动画预留设计

本次不实现路由进入退出动画，但需要保证结构已满足后续接入条件。

约束如下：

- `default-layout__view` 必须是一个独立内容层
- 背景层与内容层必须清晰分离
- 后续如果在 `view` 层外包 `Transition` 或 `TransitionGroup`，不应影响背景层稳定存在

这意味着未来页面切换时，只切内容视图，不重新创建背景壳层。

## 9. 文件改动范围

### 9.1 入口层

- Modify: `app/app.vue`

说明：

- `app/app.vue` 负责承载非视觉入口级主题注入逻辑
- 仅在默认布局页面输出 `useGlobalTheme` 的前置脚本

### 9.2 主题逻辑层

- Create: `app/composables/useGlobalTheme.ts`
- Modify or Refactor: 当前首页主题相关 composable

说明：

- 新的 `useGlobalTheme.ts` 负责颜色主题与脚本注入能力
- 当前 `useHomepageTheme.ts` 若保留，必须移除其全局颜色职责

### 9.3 布局层

- Modify: `app/layouts/default.vue`

### 9.4 页面层

- Modify: `app/pages/index.vue`

### 9.5 管理后台隔离

- Verify: `app/layouts/admin.vue`
- Verify: 所有 `layout: "admin"` 页面

说明：

- 本次不要求 `admin layout` 接入该主题系统
- 但必须验证 `admin` 页面完全不输出前台主题注入链路

## 10. 错误处理与兼容性

### 10.1 主题注入失败兜底

若前置脚本未成功执行，最低要求是前台页面仍可渲染，不因主题系统失败导致页面不可用。

因此实现阶段需要确保：

- 前台布局与页面存在合理的基础兜底颜色
- 主题变量缺失时页面仍能保持可访问

### 10.2 首屏一致性

本次主题基于浏览器本地时间生成，而不是服务端时间生成。

因此本次首屏一致性的关键要求是：

- 主题脚本必须足够早执行
- 不应出现明显的“先渲染一套前台主题，再立刻切到另一套主题”的闪动

### 10.3 admin 隔离

必须保证：

- `admin` 页面不输出 `useGlobalTheme` 的前置脚本
- `admin` 页面不依赖 `--app-theme-*`
- 前台主题实现不会改变现有 `admin` 布局和组件行为

## 11. 测试与验证

### 11.1 视觉验证

必须验证以下结果：

- 首页首屏视觉与改造前保持一致
- 不同本地时间段下，背景、文本、CTA、光晕颜色仍正确
- 默认布局的 frame 与安全区表现不偏移

### 11.2 结构验证

必须验证以下结果：

- 默认布局已形成“背景层 + view 层”的清晰分层
- 首页不再承接背景壳层职责
- 默认布局下其他页面具备直接复用主题与背景的条件

### 11.3 注入验证

必须验证以下结果：

- 默认布局页面的 SSR 输出中包含前台主题前置脚本
- 浏览器在 hydration 前可完成主题变量写入
- `default layout` 本身只消费变量，不负责注入逻辑

### 11.4 admin 隔离验证

必须验证以下结果：

- `admin` 页面 SSR 输出中不包含前台主题前置脚本
- `admin` 页面首屏和交互不受前台主题实现影响
- 从前台切到 `admin`、再切回前台时，页面行为保持稳定

### 11.5 质量门槛

实现阶段完成后，必须执行：

```bash
bun quality-check
```

并且不允许出现任何报错。

## 12. 非目标

本次方案不包括：

- 新增或重构页面进入退出动画实现
- 调整首页文案策略本身
- 为 admin 引入同一套主题系统
- 为主题系统内建统一过渡动画机制
- 对与本次主题壳层改造无关的前台页面做无关重构
