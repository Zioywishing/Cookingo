# Cookingo B 端 IAM 底座需求方案

## 1. 目标

本方案用于定义 Cookingo 第一版 B 端底座的需求范围。

本次目标不是实现业务后台功能，而是先建立一套可持续扩展的后台基础能力，覆盖：

- 后台登录与会话
- 用户管理
- 角色管理
- 页面与菜单级权限控制
- 登录日志
- 操作审计

本方案同时覆盖前端与后端的交付边界，但不在这一版需求方案中锁定具体技术选型。

## 2. 适用范围

本次方案适用于项目内部管理后台。

确认范围如下：

- 后台为单组织内部后台
- 用户通过账号密码登录
- 系统首次部署后初始化一个超级管理员
- 后续账号仅允许由管理员创建
- 角色采用完全自定义模式
- 权限只做到页面和菜单级
- 第一版包含安全运维能力

本次方案明确不包含：

- 多组织、多租户、组织树
- 自助注册、邀请注册、员工申请开通
- 企业 SSO、短信登录、邮箱验证码、2FA
- 动作级、字段级、数据范围级权限
- 报表中心、消息中心、审批流
- 菜谱、分类、内容审核等业务后台模块

## 3. 设计原则

### 3.1 先建立后台底座，再接业务模块

第一版 B 端只解决“谁能进入后台、谁能看到什么页面、谁做过什么操作”这三个基础问题，不提前掺入业务管理功能。

### 3.2 权限模型先做窄但做对

第一版权限单位统一为后台页面访问权。前端菜单展示与页面访问控制、后端接口访问控制都围绕同一套权限编码工作。

### 3.3 单组织模型优先

后台账号、角色、权限都以单组织内部使用为前提，不为未来的多租户需求提前引入复杂抽象。

### 3.4 安全能力必须形成闭环

登录、退出、禁用账号、重置密码、登录日志、操作审计必须能互相配合，避免只做 UI 骨架而没有真实安全约束。

### 3.5 后台页面和后台接口边界清晰

后台前端统一归入 `/admin` 域，后台接口统一归入 `/admin` API 域，避免与 C 端功能接口混杂。

## 4. 系统边界

### 4.1 前端边界

第一版后台前端需要提供以下能力：

- 登录页
- 后台布局框架
- 当前登录状态恢复
- 后台导航菜单
- 页面访问拦截
- 用户管理页
- 角色管理页
- 登录日志页
- 操作审计页

### 4.2 后端边界

第一版后台后端需要提供以下能力：

- 初始化超级管理员
- 登录、退出、当前会话校验
- 用户创建、查询、编辑、启用、禁用、密码重置
- 角色创建、查询、编辑、删除、分配权限
- 查询权限点列表
- 登录日志记录与查询
- 操作审计记录与查询

### 4.3 数据边界

数据库只承载后台 IAM 与安全运维数据，不承载本次未实现的业务后台数据。

本次需覆盖的数据域包括：

- 后台用户
- 后台角色
- 页面权限点
- 用户角色关系
- 角色权限关系
- 会话
- 登录日志
- 操作审计

## 5. 核心角色与权限模型

### 5.1 用户

后台用户表示一个可登录管理后台的内部账号。

用户最小信息范围包括：

- 登录标识
- 展示名称
- 密码摘要
- 状态
- 最近登录时间
- 创建时间与更新时间

第一版用户状态只保留最小集合：

- `active`
- `disabled`

### 5.2 角色

角色是权限集合的承载体。

本次明确采用完全自定义角色模式：

- 系统初始化时只保证存在超级管理员能力
- 其余角色均由后台管理员自行创建
- 角色可配置名称、稳定标识、描述和权限集合

### 5.3 权限点

第一版权限点即后台页面访问权限。

第一版内置以下基础权限点：

- `admin.dashboard`
- `admin.users`
- `admin.roles`
- `admin.login-logs`
- `admin.audit-logs`

权限点作为系统内置能力存在，不开放为任意自定义页面资源。

### 5.4 关系模型

关系主线为：

`用户 -> 用户角色关系 -> 角色 -> 角色权限关系 -> 页面权限点`

说明如下：

- 一个用户可绑定多个角色
- 一个角色可绑定多个页面权限
- 当前用户的最终权限集合由其全部角色合并得出

### 5.5 第一版权限边界

第一版只做到页面和菜单级控制。

这意味着：

- 拥有 `admin.users` 的用户可以访问用户管理页面
- 用户管理相关接口也统一要求该权限
- 不再额外区分“可看用户页但不可重置密码”这类动作级细分能力

动作级权限作为后续扩展方向保留，但不进入本次交付范围。

## 6. 关键业务流程

### 6.1 初始化超级管理员

系统首次部署后必须提供初始化入口，用于创建首个超级管理员。

约束如下：

- 初始化只允许执行一次
- 一旦系统中已存在后台用户，则不可再次初始化
- 超级管理员默认拥有全部后台页面权限

### 6.2 登录

登录方式为账号密码登录。

登录行为要求：

- 使用统一登录入口
- 校验账号状态与密码
- 登录成功后建立会话
- 登录失败时返回统一失败口径，不泄露账号是否存在
- 成功与失败都记录登录日志

### 6.3 会话恢复与退出

前端刷新后应可通过当前会话接口恢复登录态。

退出要求：

- 用户主动退出时销毁当前会话
- 前端清空登录状态并返回登录页

### 6.4 页面访问控制

页面访问控制由前后端共同完成。

前端职责：

- 根据当前用户权限生成菜单
- 对后台页面访问做前端拦截

后端职责：

- 所有后台接口按对应页面权限再做一次服务端校验

### 6.5 用户管理

具备用户管理权限的管理员可以：

- 创建后台用户
- 查看用户列表与详情
- 调整用户基础信息
- 为用户分配角色
- 启用或禁用用户
- 重置用户密码

第一版不提供物理删除用户能力，统一采用禁用机制。

### 6.6 角色管理

具备角色管理权限的管理员可以：

- 创建角色
- 编辑角色信息
- 配置角色权限
- 删除未被使用的角色

保护规则：

- 已绑定用户的角色不可直接删除
- 超级管理员能力不可被误删或削弱到无法管理系统

### 6.7 密码管理

第一版至少覆盖两类密码行为：

- 管理员重置指定用户密码
- 当前登录用户修改自己的密码

安全要求：

- 密码被重置后，该用户已有会话必须失效
- 用户被禁用后，已有会话必须失效

### 6.8 登录日志

每次登录尝试都需要记录登录日志。

最小记录要素包括：

- 账号标识
- 登录结果
- 失败原因摘要
- 请求来源信息
- 时间

### 6.9 操作审计

后台关键操作必须写入操作审计。

第一版至少覆盖：

- 初始化超级管理员
- 创建用户
- 更新用户
- 启用用户
- 禁用用户
- 重置密码
- 创建角色
- 更新角色
- 删除角色
- 调整用户角色
- 登录与退出

审计最小记录要素包括：

- 操作者
- 操作类型
- 目标对象类型
- 目标对象标识
- 操作摘要
- 时间

第一版不要求保存完整字段级 diff，但必须能回答“谁在什么时候对什么做了什么”。

## 7. 后台信息架构

第一版后台页面结构如下：

- `/admin/login`
- `/admin`
- `/admin/users`
- `/admin/users/[id]`
- `/admin/roles`
- `/admin/roles/[id]`
- `/admin/login-logs`
- `/admin/audit-logs`

页面职责如下：

- 后台首页：展示当前登录人和系统概览占位
- 用户列表页：展示账号、状态、角色、最近登录时间
- 用户详情页：展示基础信息并提供启用、禁用、重置密码、角色调整
- 角色列表页：展示角色摘要与使用情况
- 角色详情页：编辑角色与权限集合
- 登录日志页：查看登录结果记录
- 操作审计页：查看后台关键行为记录

## 8. 接口范围

第一版后台接口按能力域划分为四组：

### 8.1 认证与会话

- 初始化超级管理员
- 登录
- 退出
- 获取当前登录用户

### 8.2 用户管理

- 用户列表
- 用户详情
- 创建用户
- 更新用户基础信息
- 更新用户角色
- 启用用户
- 禁用用户
- 重置用户密码
- 用户修改自己的密码

### 8.3 角色与权限

- 角色列表
- 角色详情
- 创建角色
- 更新角色
- 删除角色
- 获取系统权限点列表

### 8.4 日志与审计

- 登录日志列表
- 操作审计列表

第一版接口以可管理、可查询、可维护为目标，不加入批量操作、导出和复杂统计。

## 9. 数据库职责边界

本次数据库方案需要支持以下需求，但本需求文档不锁定具体数据库产品与 ORM 方案。

数据库必须满足：

- 能持久化后台用户、角色、权限、关系、会话、日志、审计
- 能支持“初始化只允许一次”这类系统级约束
- 能支持“禁用用户或重置密码后立即失效已有会话”
- 能支持角色删除保护
- 能支持基于时间倒序查询登录日志和审计记录

本次数据库不需要承载：

- 多租户隔离字段
- 组织架构树
- 数据权限范围表达式
- 复杂报表模型

## 10. 错误处理与安全约束

### 10.1 登录错误口径

登录失败应使用统一错误口径，不向外暴露“账号不存在”还是“密码错误”的具体差异。

### 10.2 后台接口统一鉴权

所有后台接口都必须依赖服务端会话校验与权限校验，不允许只靠前端页面隐藏。

### 10.3 高风险操作保护

以下操作必须具备明确的服务端保护逻辑：

- 初始化超级管理员
- 禁用用户
- 重置密码
- 删除角色
- 修改高权限角色

### 10.4 可恢复优先于物理删除

第一版尽量采用禁用、失效、解除绑定等方式控制风险，不以物理删除为主要操作方式。

## 11. 非目标

以下内容不在本次实施范围：

- 菜谱后台、分类后台、审核后台、配置后台
- 自助注册与邀请开通
- 企业 SSO 与第三方身份提供方接入
- 动作级权限与数据范围权限
- 验证码、双因素认证、设备指纹
- 审计报表与安全报表
- 邮件通知与短信通知

## 12. 验收标准

满足以下条件时，可认为第一版 B 端 IAM 底座需求落地完成：

- 系统可初始化首个超级管理员且仅允许一次
- 管理员可通过账号密码登录后台
- 后台可基于当前用户权限生成菜单并限制页面访问
- 后台可完成用户创建、角色分配、启用、禁用、密码重置
- 后台可完成角色创建、编辑、权限配置与受保护删除
- 登录成功与失败均可查询到日志
- 关键后台操作均可查询到审计记录
- 用户被禁用或密码重置后，旧会话失效
- 前后端都围绕同一套页面权限编码工作

## 13. 后续技术方案输入

本方案已明确需求边界，后续技术方案需要在此基础上继续回答以下问题：

- 采用哪种数据库产品
- 采用哪种数据访问层方案
- 会话存储方式如何设计
- 密码摘要与校验策略如何实现
- 初始化与权限种子数据如何落库
- 前端状态组织与路由守卫如何实现
- 后端模块、表结构、接口文件如何组织

这些问题属于技术设计阶段，不在本次需求方案中先行锁定。
